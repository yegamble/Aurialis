## Summary

**Goal:** Add a Postman/Newman E2E suite for the Aurialis FastAPI backend that covers happy-path workflows, audio-file edge cases, security probes, and rejection of non-audio uploads — and harden the backend so non-audio rejection is real, not theatrical.

**Architecture:** A Postman v2.1 collection lives at `backend/tests/postman/aurialis-backend.postman_collection.json` with three Postman environments (local docker-compose, CI, production-smoke). A bash runner (`run-newman.sh`) wraps Newman with environment selection and folder filtering (PR CI vs nightly long-audio). Backend gains a `validation.py` module that magic-byte-sniffs uploads, enforces a 200 MB size cap, and confirms playability via `soundfile.info()` before queuing a job. CI gets a new `backend-e2e` job (PR-blocking) and a `backend-e2e-nightly` workflow (cron). Post-deploy smoke runs a tagged subset against production.

**Tech Stack:** Postman Collection v2.1, Newman CLI (`newman` npm pkg, run via `npx`), Python 3.12 + soundfile (existing), FastAPI 0.115, Docker Compose (existing), GitHub Actions.

## Scope

### In Scope

- Backend hardening: magic-byte audio sniff, 200 MB size cap, `soundfile.info()` confirmation, structured 4xx errors for non-audio / oversized / corrupted uploads.
- Postman collection covering all seven backend endpoints: `GET /health`, `POST /separate`, `POST /analyze/deep`, `GET /jobs/{id}/status`, `GET /jobs/{id}/result`, `GET /jobs/{id}/stems/{name}`, `DELETE /jobs/{id}`.
- Audio edge-case fixtures (tiny < 1 s, mono 8 kHz, mono 96 kHz, corrupted WAV header, 10-minute) generated by a Python script using existing `numpy` + `soundfile`.
- Non-audio fixtures (PDF, fake-EXE byte stub, oversized payload) generated at test time.
- Security checks: invalid model/profile values, path-traversal in `stem_name`, oversized payloads, CORS preflight from disallowed origin.
- Three execution targets: local docker-compose, GitHub Actions PR CI (long-audio excluded), GitHub Actions nightly (full suite), post-deploy production smoke (read-only happy path).
- Hybrid polling: separation enqueue-only (avoids Demucs GPU dep in CI), deep-analysis polled to completion on at least one happy-path test.
- Backend `pytest` coverage for the new validation module (≥ 80 %).

### Out of Scope

- Authentication / API keys (backend remains open for now — flagged in `Risks`).
- Rate limiting (would require Worker-level state; tracked separately).
- Replacing existing pytest suite — this complements it.
- Frontend changes (no UI implication).
- Stress / load testing (k6 or similar).
- Real-music fixtures with copyrighted content (synthetic only).

## Approach

**Chosen:** "Backend hardening + Postman collection + 3-tier execution (local / PR CI / nightly + production smoke)".

**Why:** Adding the validation layer in `backend/validation.py` makes "filter non-audio files" a real assertion the tests can hold the backend to, instead of pinning the broken pass-through behavior. Postman collection format is portable (Postman GUI for human exploration, Newman for automated runs), and folder-tagging lets us reuse one collection across PR CI (fast) and nightly (long-audio). Co-locating fixtures and the runner in `backend/tests/postman/` keeps backend test artifacts in one place. Cost: one new lightweight module (~80 LOC), one new npm dev dep (`newman`), one new CI workflow.

**Alternatives considered:**

- **Pure pytest + httpx instead of Postman.** Lighter (no new tool), but loses the GUI exploration workflow Postman offers, and forces CI to spin up Python deps just to hit HTTP endpoints. Postman/Newman is the explicit user request.
- **Add `python-magic` for libmagic-based MIME detection.** More robust than magic-byte sniff for exotic formats, but requires a system-package install (`libmagic1`) in the Dockerfile. Magic-byte sniff covers all five allowed audio formats with zero deps.
- **Tests-only, no backend changes.** Cheapest, but the user explicitly asked to "filter non-audio files" — leaving the gap open and just asserting the broken pass-through would mock that requirement.

## Context for Implementer

> Write for an implementer who has never seen the codebase.

- **Patterns to follow:**
  - FastAPI route handlers are module-level `async def _name(...)` in `backend/main.py:61-198` (lines 61-186 define handlers; 189-198 register routes). Mirror that style — keep `validation.py` plain functions that the existing handlers call.
  - Test fixtures: `backend/tests/conftest.py:21-41` shows the `synthetic_30s_wav` fixture pattern using `numpy` + `soundfile.write`. Reuse for the new fixtures.
  - Pytest markers in `backend/pyproject.toml:8-11` are `unit` and `integration`. New validation tests are `unit`.
- **Conventions:**
  - Python: uv only (`uv run pytest -q`, `uv run ruff check .`, `uv run ruff format .`). One-line docstrings; modern type hints (`list[int]`, `str | None`).
  - Postman: descriptive folder names ("Happy Path", "Audio Edge Cases", "Security", "Long Audio (Nightly)"). Tests in `tests` event scripts; reuse env vars `{{baseUrl}}`, `{{jobId}}` rather than hardcoding.
- **Key files:**
  - `backend/main.py` — route handlers; this is where validation is wired in.
  - `backend/jobs.py` — job store; tests read `partial_result` and stem metadata.
  - `backend/separation.py:16` — `VALID_MODELS = ("htdemucs", "htdemucs_6s")`.
  - `backend/main.py:19-25` — `VALID_PROFILES`.
  - `backend/main.py:28` — `ALLOWED_EXTENSIONS = {".wav",".mp3",".flac",".ogg",".m4a"}`.
  - `backend/main.py:236-246` — CORS allowlist.
  - `e2e/fixtures/generate-test-wav.mjs` — pattern reference for synthetic-audio generation (reproduce in Python for backend fixtures).
  - `docker-compose.yml` — boots backend on `localhost:8000` with `docker compose up backend`.
- **Gotchas:**
  - `soundfile.info()` only reads the header; it's cheap. But for some malformed inputs it raises `LibsndfileError`; catch broadly in `validation.py` and translate to HTTP 400.
  - The current handlers `await file.read()` *before* validation — must reorder so we read into a bounded buffer (using `file.read(MAX_SIZE + 1)` and rejecting if longer) and only persist after validation. Otherwise a 1 GB upload still fills `/tmp` before we look at it.
  - Newman exits non-zero on test failure — that's our CI signal; no extra parsing needed.
  - Postman `formdata` file uploads in v2.1 collection JSON reference local paths; the runner must resolve them relative to the collection file (Newman handles this when given `--working-dir`).
  - `htdemucs` separation requires GPU + ~2 min wall time even on a small file — never poll separation jobs to completion in CI. Tests assert only the immediate `{job_id, status}` response. Deep analysis is CPU + ~30 s on a 3-second clip — fine to poll.
  - The Cloudflare Worker (`backend/src/worker.ts`) is a transparent proxy — production smoke tests hit `https://aurialis-core.yosefgamble.com` and exercise the same endpoints.
- **Domain context:**
  - Two job types: `separation` (Demucs stems) and `deep_analysis` (sections + mastering script). Both share the same `Job` dataclass; discriminator is `job_type`.
  - "Profile" in `/analyze/deep` is one of five mastering presets (`modern_pop_polish`, `hip_hop_low_end`, `indie_warmth`, `metal_wall`, `pop_punk_air`) — these are validated server-side at `main.py:120`.
  - Job IDs are UUID4 strings. `GET /jobs/<bogus>/status` must return 404, not 422.

## Runtime Environment

- **Start command:** `docker compose up backend` (from repo root) — boots FastAPI on `localhost:8000`. First build downloads PyTorch + Demucs models (~6-8 min); subsequent builds cache.
- **Port:** 8000.
- **Health check:** `GET /health` returns `{status: "ok", gpu: bool, models: list[str]}`.
- **Restart procedure:** `docker compose restart backend` (preserves `/tmp/smart-split` job state via the `smart-split-temp` volume).
- **Production URL:** `https://aurialis-core.yosefgamble.com` (Cloudflare Worker → Container).

## Assumptions

- The current backend has zero auth on purpose (frontend is the only client today). — supported by `backend/main.py` having no auth middleware. Tasks 1, 6 depend on this.
- We deliberately rely on magic-byte sniffing only for MP3 and M4A and skip `soundfile.info()` for those formats. This is a **policy decision** (consistent rejection across libsndfile versions), not a capability claim — libsndfile ≥ 1.1.0 actually does parse MP3 headers, but coverage is platform/version-dependent and we want a single hardened rejection path. WAV / FLAC / OGG go through `soundfile.info()` because libsndfile has stable, decade-old support for them. — Task 1 depends on this.
- GitHub Actions runners can run `docker compose up backend -d` and reach the FastAPI container on `localhost:8000`. — supported by existing `.github/workflows/ci.yml:46-47` already pointing the e2e bundle at `localhost:8000`. Tasks 8, 9 depend on this.
- Newman is pinned as a `devDependency` in the **root** `package.json` (^6.2.0) and run via `pnpm exec newman` so CI uses the same version as local dev and avoids per-run `npx` downloads. — Task 7 depends on this.
- Long-audio nightly runs are tolerated to use ~5 minutes of GHA compute per night. — supported by typical project budgets; can be gated behind a workflow `inputs.run_long` if needed. Task 9 depends on this.
- Polling intervals between repeated `GET /jobs/{id}/status` requests are enforced by Newman's `--delay-request <ms>` CLI flag, NOT by `setTimeout` in Postman test scripts (Newman's vm sandbox does not pause on `setTimeout`). — Task 4 depends on this.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Backend size-cap breaks legitimate large uploads (e.g. lossless 30-min album track) | Low | Medium | 200 MB covers ~15 min stereo WAV — > 99 % of expected workflow. Surface limit in error message: `{"detail": "File exceeds 200 MB limit"}`. Document in DEPLOY.md. Verified by Task 1 unit test. |
| Validation behavior for MP3/M4A varies by libsndfile version on the deploy target | Low | Medium | Policy: validation routes WAV/FLAC/OGG through `soundfile.info()` and routes MP3/M4A through magic-byte sniff ONLY (no soundfile.info call), regardless of whether the local libsndfile happens to support MP3. This gives consistent rejection behavior across platforms. Verified by `test_validation.py` asserting (a) a real MP3 fixture is accepted via magic-byte path and (b) `soundfile.info` is never called for MP3/M4A inputs (mock + assert no-call). |
| Backend has no auth — production smoke tests inadvertently DoS the live container by polling jobs aggressively | Medium | Medium | Production smoke runs only the read-only happy path (`/health` + GET against pre-existing job IDs from a seeded fixture if available, otherwise health-only). NEVER triggers separation or analyze on prod. Verified by Task 9 production env config. |
| Demucs/madmom not present in CI containers cause `/separate` and `/analyze/deep` to error before validation | Medium | Medium | Backend hardening reorders: validate FIRST, queue SECOND. Validation rejection (4xx) precedes any ML import. CI separation tests assert only the 200 + job_id response — they don't poll, so the missing GPU never matters. |
| Newman's HTML reporter generates large artifacts that bloat CI cache | Low | Low | Use `--reporters cli,junit` only. JUnit reports are small XML. No HTML reporter. |
| `Cancelled by user` error in deep-analysis happy-path test (race between poll & cancel) | Low | Low | Happy-path deep-analysis test does NOT cancel — separate folder for cancel tests with their own short-clip fixture. |
| Newman's vm sandbox does not honor in-script `setTimeout`, so a naive poll loop hammers the backend with zero delay | Medium | High | Polling is implemented at the Newman runner level via `--delay-request <ms>` (set per-folder in `run-newman.sh`), NOT inside Postman test scripts. Verified by Task 4 DoD check that timestamps between sequential `Poll Status` requests are ≥ `pollIntervalMs`. |
| Post-validation handler exception (e.g., `start_separation` raises) leaks a temp file to `/tmp/smart-split` | Low | Medium | Handler wraps `tmp.write(data); start_separation(input_path, ...)` in try/except that calls `os.unlink(input_path)` on any exception before re-raising. Verified by `test_validation.py::test_handler_unlinks_tempfile_on_failure`. |
| `oversized.bin` (250 MB) fixture leaks across CI runs and inflates artifact storage / cache | Medium | Low | Fixture generator stores expected size + sha256 sidecar; CI runner pre-deletes `backend/tests/postman/fixtures/*.bin` and `*.wav` before regenerating. Always-run cleanup step at end of `backend-e2e` job in `ci.yml`. |

> ⚠️ Mitigations are commitments — verification checks they're implemented.

## Goal Verification

### Truths

1. `POST /separate` and `POST /analyze/deep` return `400` with a structured `detail` field when uploaded with a PDF, fake-EXE, oversized (> 200 MB), or extension-spoofed (`evil.pdf` renamed to `evil.wav` but containing PDF bytes) payload — verified by **TS-006**, **TS-007**, **TS-008** and `test_validation.py`.
2. The Postman collection has at least one passing test for every public backend endpoint EXCEPT `GET /jobs/{id}/stems/{name}` (which requires a completed separation job — out of CI's GPU budget). Stem download is covered by `test_main.py::test_download_stem_*` (path-traversal sanitization, missing-stem 404, completed-stem 200) using a seeded in-memory job fixture. Endpoints covered by Postman: `/health` (TS-001), `/separate` (TS-002, TS-005, TS-006, TS-007, TS-008, TS-009), `/analyze/deep` (TS-003, TS-005, TS-006, TS-009, TS-012), `/jobs/{id}/status` (TS-002, TS-003), `/jobs/{id}/result` (TS-003, TS-012), `/jobs/{id}` DELETE (TS-004). Stem download endpoint covered by pytest only — see TS-013.
3. `pnpm exec newman run backend/tests/postman/aurialis-backend.postman_collection.json -e backend/tests/postman/environments/local.postman_environment.json --folder "PR Suite"` exits 0 against a freshly booted `docker compose up backend` — verified by **TS-001** end-to-end.
4. `.github/workflows/ci.yml` has a `backend-e2e` job that runs the PR Suite folder and blocks merge on failure — verified by reading the workflow file and seeing the new job in the next PR's checks list.
5. `.github/workflows/backend-e2e-nightly.yml` exists, runs on cron, and includes the `Long Audio (Nightly)` folder — verified by reading the workflow file.
6. The backend rejects oversized uploads structurally — the request returns 413 with `{"detail": "File exceeds 200 MB limit"}` AND no temp file is created in `/tmp/smart-split` for the rejected upload. Verified by **TS-007** (response code + post-request listing of `/tmp/smart-split` is unchanged) and `test_validation.py::test_oversized_rejected_without_persisting_tempfile` (mocks `UploadFile.read` to simulate oversized data and asserts `tempfile.NamedTemporaryFile` is never called). Time-based assertions are intentionally omitted — Starlette's `UploadFile` spools to disk before the handler runs, so response time reflects network speed, not validation efficiency.

### TS-007: Reject Oversized Uploads (Bounded Persistence)
**Priority:** Critical (Security)
**Preconditions:** `oversized.bin` (250 MB) generated at runtime by `generate-fixtures.py`; no time-based assertion (Starlette spools the body before the handler runs, so request duration reflects network speed not validation efficiency).
**Mapped Tasks:** Task 1, Task 6

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | `GET {{baseUrl}}/admin/list-tmp` (test-only debug endpoint) **OR** snapshot `/tmp/smart-split` listing via a sidecar pytest helper before the test | Baseline list captured |
| 2 | `POST /separate` with `oversized.bin` (250 MB) | 413; `detail` mentions size limit |
| 3 | Re-snapshot `/tmp/smart-split` listing | Listing is unchanged from step 1 (no new file from the rejected upload) |

> Note: Steps 1 and 3 require either a small test-only endpoint behind an env-flag (`AURIALIS_TEST_HOOKS=1`) or a sidecar pytest helper invoked from a Newman pre-request script via `pm.sendRequest` against the host filesystem. Implementer chooses; the structural assertion ("temp file not persisted") is the actual contract — see `test_validation.py::test_oversized_rejected_without_persisting_tempfile` for the pure-pytest equivalent.

### TS-010: Path Traversal in stem_name (Pytest)

This is intentionally NOT a Postman scenario because exercising a real path-traversal requires a completed job, and CI has no GPU to complete a separation.
**Mapped Tasks:** Task 6 (pytest, NOT Postman)

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | `_download_stem(job_id, stem_name="../../etc/passwd")` | Raises `HTTPException(status_code=404)` — stem name not in `[s.name for s in job.stems]` |
| 2 | `_download_stem(job_id, stem_name="vocals/../../etc/passwd")` | Raises `HTTPException(status_code=404)` |
| 3 | `_download_stem(job_id, stem_name="vocals")` | Returns `FileResponse` for the legitimate stem path |

### Task 1: Backend audio-validation module + handler integration

**Objective:** Add `backend/validation.py` exposing `validate_audio_upload(file: UploadFile, max_bytes: int) -> bytes` that streams up to `max_bytes + 1` bytes, magic-byte-sniffs the prefix, optionally calls `soundfile.info()` for WAV/FLAC/OGG, and raises `HTTPException(400|413)` on rejection. Wire into `_separate` and `_analyze_deep`.
**Dependencies:** None
**Mapped Scenarios:** TS-006, TS-007, TS-008

**Files:**

- Create: `backend/validation.py`
- Modify: `backend/main.py` (call validator before `tempfile.NamedTemporaryFile`; pass content bytes to writer to avoid double-read)
- Test: `backend/tests/test_validation.py`

**Key Decisions / Notes:**

- `MAX_UPLOAD_BYTES = 200 * 1024 * 1024`. Read with `await file.read(MAX_UPLOAD_BYTES + 1)`; if `len(data) > MAX_UPLOAD_BYTES`, raise `HTTPException(413, "File exceeds 200 MB limit")`.
- Magic-byte signatures: WAV = `RIFF....WAVE` (bytes 0-3 + 8-11), FLAC = `fLaC` (bytes 0-3), OGG = `OggS` (bytes 0-3), MP3 = `ID3` (bytes 0-2) OR `0xFF 0xFB|0xF3|0xF2` (frame sync, bytes 0-1), M4A = bytes 4-7 = `ftyp` (any brand).
- For WAV/FLAC/OGG only: write to an `io.BytesIO` and call `soundfile.info(buf)` — if it raises `LibsndfileError`, return 400 "Could not parse audio". For MP3/M4A **deliberately skip** the soundfile step (policy choice — libsndfile MP3/M4A support varies by version/platform; we want consistent rejection across deploys). Magic-byte check is the only audio-content gate for MP3/M4A.
- Refactor `main.py:_separate` and `_analyze_deep` to call `data = await validate_audio_upload(file, MAX_UPLOAD_BYTES)` BEFORE the `tempfile.NamedTemporaryFile` block, then `tmp.write(data)` (avoiding the second `await file.read()`).
- **Temp-file cleanup on post-validation failure.** Wrap the `tmp.write(data); input_path = tmp.name` + `start_separation(input_path, ...)` block in `try/except` — on any exception, call `os.unlink(input_path)` (with `errno.ENOENT` swallow) before re-raising. Same pattern in `_analyze_deep`. Prevents leaks if `start_separation` or `start_deep_analysis` raises after temp creation.
- Keep `ALLOWED_EXTENSIONS` for picking the temp suffix only (cosmetic) — validation is content-based, not extension-based.
- Performance: validation runs synchronously in the request handler (FastAPI awaits the read). Magic-byte check is O(12 bytes) — trivial. `soundfile.info()` reads only the header — bounded. No caching needed.

- [ ] `_separate` and `_analyze_deep` reject non-audio with 400 BEFORE writing temp file.
- [ ] `_separate` and `_analyze_deep` reject > 200 MB with 413 BEFORE writing temp file.
- [ ] PDF, EXE, JPG, and arbitrary garbage all return 400.
- [ ] Spoofed file (`.wav` extension, PDF bytes) returns 400.
- [ ] Corrupted WAV (RIFF header but invalid data chunk) returns 400.
- [ ] All five legitimate audio formats (WAV, FLAC, OGG, MP3, M4A) pass validation.

### Task 4: Postman folder — Happy Path (TS-001..TS-004)

**Objective:** Implement Health, separation enqueue, deep-analysis full round-trip (with polling), and cancel scenarios.
**Dependencies:** Task 1, Task 2, Task 3
**Mapped Scenarios:** TS-001, TS-002, TS-003, TS-004

**Files:**

- Modify: `backend/tests/postman/aurialis-backend.postman_collection.json`

**Key Decisions / Notes:**

- **Polling delay is enforced by Newman, not Postman scripts.** Postman/Newman's vm sandbox does NOT pause execution on in-script `setTimeout` (the callback is registered but the script returns immediately). The runner instead invokes Newman with `--delay-request 2000` for the "Happy Path" folder and `--delay-request 5000` for "Long Audio (Nightly)". Polling logic in the script only decides whether to repeat (continue polling) or move on (stop polling) — it does NOT control the delay.
- Polling control flow in the deep-analysis test:
  ```js
  const status = pm.response.json().status;
  if (status === 'done') {
    pm.environment.unset('pollCount');
    pm.execution.setNextRequest('Get Result');
  } else if (status === 'error') {
    pm.test('job did not error', () => pm.expect.fail('Job errored: ' + pm.response.json().error));
    pm.execution.setNextRequest(null); // halt this folder
  } else {
    const count = parseInt(pm.environment.get('pollCount') || '0') + 1;
    pm.environment.set('pollCount', count);
    if (count >= parseInt(pm.environment.get('pollMaxIterations'))) {
      pm.test('poll did not time out', () => pm.expect.fail('Poll timeout after ' + count + ' iterations'));
      pm.execution.setNextRequest(null);
    } else {
      pm.execution.setNextRequest('Poll Status'); // Newman's --delay-request governs the wait
    }
  }
  ```
- Runner script must accept a `--delay-request <ms>` flag and forward it to Newman. The "Happy Path" folder run uses 2000 ms; "Long Audio (Nightly)" uses 5000 ms.
- Cancel test: queue → DELETE → assert `cancelled: true` → DELETE again → assert idempotent.
- Save `job_id` to env var inside test scripts: `pm.environment.set('jobId', pm.response.json().job_id)`.
